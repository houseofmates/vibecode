# vibecode Amnesia-Proof Automation

This file documents the layers of protection ensuring vibecode customizations
survive git operations and you never lose the upstream merge procedure.

## Layer 1: Skill (Hermes Agent long-term memory)

The skill `vibecode-upstream-merge` in ~/.hermes/skills/coding/vibecode-upstream-merge/
contains the complete merge protocol with exact commands. Anytime you say
"sync vibecode" or "merge upstream vibecode", the agent auto-loads this skill.

## Layer 2: Git hook (post-merge)

File: `.githooks/post-merge`
Runs `re-apply-vibecode-customizations.sh` after every git pull/merge.
Configured via: `git config core.hooksPath .githooks`

If a merge silently destroys vibecode customizations, this hook catches it
immediately before you push.

## Layer 3: Verification script

File: `re-apply-vibecode-customizations.sh`
Idempotent script that checks:
- All vibecode-only files exist
- server.py has CORS, warmup, cron cleanup, POST pre-read, large queue
- routes.py has swarm imports, wiki_memory imports, sudo_password imports
- routes.py has vibecode route handlers
- All Python files compile clean
- No merge conflict markers

Exit 0 = all good. Exit 1 = issues found.

Run anytime with: `bash re-apply-vibecode-customizations.sh`

## Layer 4: Cron job (upstream checker)

Runs every 6 hours via Hermes cron: checks if upstream has new commits
since the last merge (v0.50.68 / 6c5911a). Notifies if 10+ new commits.

## Layer 5: Backup before any merge

The merge protocol always:
1. Creates backup branch: `vibecode-pre-merge-YYYYMMDD`
2. Tags: `vibecode-pre-merge-YYYYMMDD`
3. Pushes both to origin

## Layer 6: This file

`.hermes-update-automation.md` in the repo root documents the entire
amnesia-proof system. If all else fails, read this file and start from here.
